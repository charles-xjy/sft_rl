## 1. analysis 过度模板化

**现象**:首版 5000 条数据,100% 样本都是 `<analysis>` 固定 3 步(定位/观察/判断)+ `<answer>`;`<analysis>` 步数分布 `{3: 5000}`,开头 100% 是"1. 定位：关注图像…"。

**根因**:`ANSWER_GENERATION_PROMPT` 硬性规定"必须固定为 3 步"。

**危害**:模型会把"固定作文模板"当成必答格式——连"图里有没有船"这种只需答"是"的问题也先写三步分析,导致**答对了但废话多**,这正是 VL 模型最被诟病的点。本质是回答熵塌缩。

**解决**:
- analysis 步数改为**自适应 1-3 步**,简单题 1 步即可,禁止凑步数注水。
- 新增 `ANSWER_DIRECT_PROMPT`(直答、无 analysis),通过 `--direct-ratio`(默认 0.3)混入约 30% 直答样本,让模型学会"**按问题选择回答方式**"。
- `validator` 支持 analysis / direct 双模式(`expect_analysis` 参数),否则直答样本会被全部误杀。
- `spatial_relation` / `unanswerable` / `ambiguous` 需要证据链,固定走 analysis。

**验证**(新数据 3461 条):answer_mode = analysis 77.9% / direct 22.1%;analysis 开头最高仅 1.5%,不再是固定起手式。

> 备注:direct 实测 22% 而非 30%,因为 spatial/unanswerable/ambiguous 被强制走 analysis,剩余约 74% 才按 0.3 抽直答(0.3×74%≈22%),符合设计。想要真 30% 把 `--direct-ratio` 提到 ~0.4。

## 2. 描述成为单点真相源

**现象**:原流程为 `图→描述→问题→答案`,且答案生成用 `图 + 描述 + 问题`,对描述强依赖。

**根因**:Phase1 描述一旦出错(如"道路"写成"河流"),问题和答案都会跟着错,形成"错描述→错问题→错答案"的自洽闭环。

**解决**:Phase2 / Phase3 的 prompt 均改为明确**"描述仅供参考、可能有误,一切以图像为准,冲突时以图为准"**;描述从"真相源"降级为"弱参考"。

**遗留**:问题侧仍是描述驱动(见问题 8 的 yes 偏置),未完全解耦。

---

## 3. counting 占比过高、噪声大

**现象**:首版各题型自由生成,实测 counting ≈ 20%。

**根因**:出题完全交给模型,无配比控制;而遥感计数受遮挡、分辨率、边界模糊影响极大,"17 栋"实际可能是"15~20 栋",噪声远高于 existence/location。

**解决**:
- 目标配比中把 counting 压到 **5%**(`src/sampling.py:SCHEDULE_DIST`)。
- 生成时软配额(见问题 5)+ 发布时降采样(`scratch/learn/02_template_health.py --rebalance`)双重控制。

**验证**:新数据 counting = 5.3%,命中目标。

---

## 4. 缺少拒答样本

**现象**:首版 100% 样本都有确定答案,无"无法确认"类样本。

**根因**:未主动构造拒答样本。

**危害**:模型学到"必须回答",遇到看不出的就硬编 → 这是 VL 微调最易翻车的幻觉点。

**解决**:新增两类样本(合计目标 ~6%):
- `unanswerable`:仅凭图像无法确认(用途/材质年代/权属等),答"无法根据图像确认…"。
- `ambiguous`:存在多个合理解释(如"仓库还是厂房?"),答"无法准确区分,可能是…也可能是…"。
- `validator` 校验这两类必须给出拒答/多解表达。

**验证**:新数据拒答类占比 5.8%(落在 5-10% 目标带),表达正确率 100%。

---

## 5. 问题分布塌缩风险

**现象/隐患**:Phase2 自由出题,长期运行易塌缩成"是否有建筑/道路/植被"刷屏(Question Distribution Collapse),且类型比例漂移。这比幻觉更隐蔽——5000 条看着多,实际是同一种任务的 5000 次重复。

**解决**:
- **生成时软配额**(`src/sampling.py`):按目标分布给每张图确定性分配一份"建议类型清单"注入 prompt,图像足够多时聚合分布收敛到目标。软约束,模型可对不适合的图跳过某类型。
- **反塌缩 prompt**:明确"同类问题不要相同句式起手或问同一对象"。
- **监控**:`scratch/learn/02_template_health.py` 报告 question_type 归一化熵(<0.7 红线)、问题前缀 Top-N(前 3 合计 >60% 判塌缩)。

**验证**:新数据类型配比贴合目标、归一化熵 0.85;问题前缀前 3 合计仅 20.2%(远低于 60% 红线),**未塌缩**。

## 6. 同一教师模型自出自答

**现象/隐患**:三阶段默认同一教师模型。问题 2 的"以图为准"只打破了部分闭环,问题/答案的风格与偏好仍同源,信息增量偏低。

**解决方向(待办)**:脚本已支持各阶段独立传 `--base-url` / `--model`。数据量做到 50k+ 时建议换不同教师(如问题用模型 A、答案用模型 B)提升监督多样性。属于运行层配置,无需改代码。

**状态**:⏳ 暂未实施,规模上来后再做。

## 7. 9% 样本 format_fail(漏 `</answer>` 标签)

**现象**:新数据 3461 条中 314 条(9.1%)被自动质检判 `format_fail`,体检报告里表现为 9.1% 的"空答案开头"。

**定位过程**:
- 抽样看 format_fail 内容:`<analysis>…</analysis>\n<answer>\n是`——answer 内容完整正确,**只是缺收尾的 `</answer>`**。
- 按类型分布:existence 293 / spatial 10 / location 9 / attribute 2;answer 尾部 298 个是"是"。
- `enable_thinking:False` 已生效(`vlm_client.py`),排除思考链吃预算。

**根因**(两个叠加):
1. **服务端 `--max-model-len 4096` 太小**:对带图 VLM,图像 token(常 1000~3000+)+ prompt 已占大半,而脚本请求 `max_tokens=4096` ≥ 总上限,输出空间被挤压甚至截断。已将服务端调到 8192、脚本 `max_tokens` 调到 2048。
2. **模型对超短答案(existence 的"是")倾向答完即发结束符(EOS),漏掉 `</answer>`**——chat/instruct 模型做结构化标签输出的通病,非 prompt 格式错误。

**危害**:这 314 条被写进了输出文件(生成脚本不管 is_valid 都写盘),且降采样脚本当前**不过滤** format_fail,会漏进发布集。

**解决方案(已设计,暂未实施——按决定先不救)**:
- 容错补标签:`<answer>` 存在但缺 `</answer>` 时自动补回(答案本身正确,可安全救回绝大多数)。
- 降采样时排除仍为 format_fail 的样本。
- 可选 prompt 微调:existence 答成完整短句而非光一个"是",降低提前 EOS 概率。

**状态**:🔍 根因已确认,修复方案就绪,按当前决定**暂不修复**。

## 8. existence 答案 yes 偏置

**现象**:新数据 existence 有效样本中,`是/有` 869 条 vs `否/无` 8 条 —— **yes:no ≈ 99:1**(算上同为"是"的 format_fail 更极端)。

**根因**:问题从描述生成,而描述只列图里**有**的东西,于是 existence 只会问"有没有[图里确实有的地物]",答案几乎全是"是"。这是问题 2(描述耦合)在**问题侧**的残留。

**危害**:existence 占全数据 33.8%。模型会学到捷径"看到 existence 就答是",不看图也能拿高分;推理时对图中不存在的东西也倾向答"有"=**凭空幻觉**,正是项目要压制的遥感幻觉。

**解决**:`QUESTION_GENERATION_PROMPT` 要求 existence **正反均衡**——约一半针对图中确实存在的地物提问(答是),另一半针对该场景合理但本图不存在的地物提问(如农田图问船舶/高楼,答否)。
- 安全性:即便 Phase2 猜错(问的东西其实图里有),Phase3 以图为准答,**不会引入错答**,只把 yes/no 拉平。

**状态**:🛠 已改 prompt(commit `1d1f0c1`),**需删除旧 `02_questions.jsonl` / `03_answers_sft.jsonl` 后重跑 Phase2+3** 才生效;重跑后用 yes/no 命令复测是否拉到 ~5:5。
